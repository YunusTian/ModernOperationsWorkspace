## 1. 缺陷背景

`release.yml` 的 `smoke` job 采用 `binaries-<target>-<arch>` artifact，只下载"当前 target/arch"的 tar.gz。历史的 [release-smoke.ps1](../scripts/release-smoke.ps1) 在派生本地 catalog 时对 `entries[].versions[].platforms[]` 的每一项都调用 `Resolve-Path -LiteralPath`；当 catalog 里存在其它平台（linux/darwin）的条目时，本地文件不存在，`Resolve-Path` 直接抛错，导致 Windows Phase 2 catalog install smoke 从未成功过。Bash 版本用 `os.path.abspath` 得以侥幸通过，两端语义并不一致。

## 2. 修复

- [x] 抽出可测试的纯 JSON 变换：[scripts/release-smoke-lib.ps1](../scripts/release-smoke-lib.ps1) 提供 `ConvertTo-LocalCatalog -Catalog -ArtifactDir -Target -Arch`
    - 按 `Target/Arch` 过滤 `versions[].platforms`
    - 磁盘上缺失的产物直接跳过（对齐 [release-smoke.sh](../scripts/release-smoke.sh) 宽松语义）
    - 匹配到且存在的条目才改写为 `file:///` URL
- [x] [scripts/release-smoke.ps1](../scripts/release-smoke.ps1) 在 Phase 2 改为调用 `ConvertTo-LocalCatalog`，其它行为保持不变（同一临时目录、同一 `plugin catalog refresh → search → install ssh → uninstall --purge` 链路）
- [x] 无 Bash 侧改动（[release-smoke.sh](../scripts/release-smoke.sh) 语义已经是宽松过滤，无需修改）
