## 0. 摘要（TL;DR）

v0.6.0 是 Workflow 2.0 的**第一个子版本**，**只做增量语法**，不动执行历史后端、不引入版本化和审批、不接触触发器。目标是让 v0.3 的 YAML 能表达三类新场景：

1. **并发上限与嵌套并行组** —— `parallel_limit: N` + 单层嵌套 `parallel_group:`
2. **单 step target 覆盖 + 子工作流调用** —— `step.target: <id>` 与 `step.workflow: <ref>`
3. **参数 JSON Schema 化 + 幂等键** —— `inputs.schema:` 与 `workflow.idempotency_key:`

**核心承诺**：所有新字段**均可选**。旧 v0.3 YAML **一字不动**继续跑；YAML 严格模式 (`KnownFields(true)`) 保持不变。

**非目标（推给 v0.6.1+）**：Draft/Published/Deprecated 版本化 · 手工审批节点 · Dry-run / Plan · 定时/Webhook 触发 · SQLite 结构化历史 · 通知插件。

---

## 1. 目标与非目标

### 1.1 目标

- 让"跨机器 fan-out"、"上百主机滚动"、"引用共享 recipe"三类真实场景可以直接落 YAML，不再靠外部脚本组织
- 用 JSON Schema 替代 `inputs[].type` 的枚举窄口子，与 v0.5.2 的 `settingsSchema` 编译器 [core/plugin/settings/schema.go](../core/plugin/settings/schema.go) 复用
- 提供**幂等键**给未来的定时/Webhook 触发（v0.6.3）与 Runbook 重放（v0.9 AI Plan）铺路
- **不改破坏性**：所有 v0.3 workflow / v0.4/v0.5 交付物 100% 通过

### 1.2 非目标（v0.6.0 不做）

| 项 | 归属版本 | 理由 |
|---|---|---|
| Workflow 版本化（Draft → Published → Deprecated） | v0.6.1 | 需要 ID + 不可变版本号 + 存储层改造，独立成子版本 |
| Dry-run / Plan / 审批节点 | v0.6.3 | 依赖版本化落地 |
| SQLite 结构化历史 | v0.6.2 | 单独 tag 好回退，避免与 DSL 变更同 tag |
| 触发器（定时 / Webhook / 通知插件） | v0.6.3 | Provider 抽象 + 恢复语义需要独立设计 |
| `parallel: continue`（组内失败继续其它兄弟） | v0.7 | 与 rollback / on_failure.strategy 联动，先攒 v0.6 使用反馈 |
| 完整表达式重构（`${}` 与 `when` 统一语法） | 长期 | 兼容性代价太大，等 v1.0 前评估 |
| 每 step 独立 Provider / 独立凭据 | v0.7 | 与 Connection Manager 深耦合 |

## 3. DSL 增量（五项）

### 3.1 `parallel_limit: N` 并发上限

**动机**：v0.3 P5 的 `parallel: true` 是"整组同时并发"。跑 100 台机器巡检时，同时握手 100 个 SSH 会打爆连接池 / 目标主机的 sshd MaxStartups。

**YAML**：

```yaml
steps:
  - id: probe_all
    parallel: true
    parallel_limit: 8         # 该并发组同时最多 8 个 in-flight
    command: ssh.exec
    params: { cmd: "uptime" }
  - id: probe_all_2
    parallel: true
    command: ssh.exec
    params: { cmd: "df -h" }
  # …连续 20 个 parallel: true 归为同一组，同时最多 8 个
```

**语义**：

- 声明位置：**组内任一 step** 都可标 `parallel_limit`，取组内的**最大值**（0 = 无上限）
    - 拒绝方案：把 `parallel_limit` 放到"组"级别 —— 无处安放，YAML 里没有"组"这个显式节点
    - 拒绝方案：只能第一个 step 标 —— 用户容易漏写而不自知
- `parallel_limit: 0` 等价于"缺省"，与 v0.3 行为完全一致
- `parallel_limit > 0` 且 `!parallel` → **Validate 报错** `PARALLEL_LIMIT_WITHOUT_PARALLEL`
- fail-fast 语义不变：任一 in-flight step 失败仍然取消同组 pending step 的 ctx
- 硬上限 `parallel_limit ≤ 512`（防 YAML 手误）

**实现草案**（[core/workflow/runner.go](../core/workflow/runner.go)）：

- 新增 `groupSemaphore = make(chan struct{}, limit)`；每个组独立
- worker goroutine 起 `sem <- struct{}{}` 才能进入 `runStep`；退出时 `<-sem`
- fail-fast：cancel 派生 ctx；已获得 slot 的 step 会通过 ctx 感知；pending 的 goroutine 直接 return

### 3.2 单层嵌套并行组（`parallel_group:`）

**动机**：滚动发布场景要"并发 batch，batch 内串行"（例：3 组 × 每组 5 台机器串行），当前 `parallel: true` 只能表达一层。

**YAML**：

```yaml
steps:
  - id: rolling
    parallel_group:                       # 显式声明"这是一个组"
      parallel_limit: 3                   # 3 个 batch 并发
      branches:                           # 每个 branch 内部串行
        - name: batch-a
          steps:
            - id: a_stop
              command: ssh.exec
              params: { cmd: "systemctl stop app" }
            - id: a_deploy
              command: ssh.exec
              params: { cmd: "cp new /opt/app" }
        - name: batch-b
          steps: [...]
        - name: batch-c
          steps: [...]
```

**语义**：

- `parallel_group.branches[]` 内每条 branch 是一个**顺序段**（不能再嵌套 `parallel_group` —— v0.6.0 只支持单层）
- branch 内 step 之间正常按声明顺序执行；不同 branch 之间并发
- `branches[i].steps[j]` 的 step **完整继承** v0.3 语义：`when / retry / compensate / timeout` 全部可用
- branch 之间**不允许**引用彼此 step 的 `out`（复用 v0.3 §7.4.5 的 [findSiblingRef](../core/workflow/workflow.go#L442) 静态检查，扩到 branch 边界）
- `parallel_group.parallel_limit = 0` → 全部 branch 并发；`> 0` → 限流
- **不允许**在 branch 内部再放 `parallel: true`（组中组的组 —— v0.7 再讨论）
- 组级 `on_failure` **不引入**：整组的 fail-fast 语义与 v0.3 P5 一致；补偿仍走顶层 `on_failure.rollback`

**为什么选嵌套结构而不是"标签归组"**：

- 标签归组（例：`parallel_group_id: "batch"`）在 YAML 里不直观，PR review 时看不出"哪几个 step 是同一组"
- 嵌套结构在 diff 里一目了然；且 Validate 逻辑天然按 tree 递归
- 权衡：写起来行数多一点。可以后续加 shorthand，但 v0.6.0 只做最保守的显式形态

### 3.3 单 step `target` 覆盖

**动机**：Workflow 主 target 是 `srv1`，但其中一步要跨机器（例：从 `srv1` 拉数据、往 `db1` 灌数据）；当前只能拆两个 Workflow。

**YAML**：

```yaml
workflow:
  id: cross-host.migrate
  steps:
    - id: dump
      command: ssh.exec
      params: { cmd: "mysqldump --all-databases > /tmp/dump.sql" }
      # target 缺省，继承 workflow.run 的 --target=srv1

    - id: scp
      command: ssh.exec
      target: db1                # 显式覆盖到 db1
      params: { cmd: "scp srv1:/tmp/dump.sql /var/lib/backup/" }

    - id: restore
      command: ssh.exec
      target: db1
      params: { cmd: "mysql < /var/lib/backup/dump.sql" }
```

**语义**：

- `step.target` 为空 → 继承 `RunnerOptions.TargetID`（v0.3 现状）
- `step.target` 非空 → 该 step 使用指定 target；同一 Workflow 可能出现 3–5 个 target
- `parallel_group` 的每个 branch 允许有**独立的默认 target**（[§3.2 branches[].target](#32-单层嵌套并行组parallel_group)）
- Target 解析在 `CommandRunOptions.TargetID` 层做，Runner 只是选值
- **不做**：跨 target 的 out 共享做特殊处理（已有的 `steps.<id>.out.*` 语义与 target 无关）
- Validate：`target: ""` 视为未设置；不解析 target 存在性（那是 Command Engine 的事，避免 workflow 校验依赖 config）

**扩展 `branches[].target`**：

```yaml
parallel_group:
  branches:
    - name: batch-a
      target: srv1              # branch 级默认，可被 branch 内 step 再覆盖
      steps: [...]
    - name: batch-b
      target: srv2
      steps: [...]
```

优先级：`step.target > branch.target > workflow (RunnerOptions.TargetID)`。

### 3.4 子工作流调用（`step.workflow:`）

**动机**：把复用度高的 5–10 步流程抽成独立 workflow 文件（例：`recipes/deploy-web.yaml`），避免在多个 workflow 里 copy-paste。

**YAML**：

```yaml
workflow:
  id: rollout.web
  inputs:
    - name: version
      schema: { type: string, pattern: "^v[0-9]+\\.[0-9]+\\.[0-9]+$" }
  steps:
    - id: canary
      workflow: ./deploy-web.yaml       # 相对当前 workflow 文件
      target: canary1
      inputs:
        version: "${inputs.version}"
        service: "web"

    - id: rollout
      parallel_group:
        parallel_limit: 3
        branches:
          - name: batch-a
            target: web-a
            steps:
              - id: deploy_a
                workflow: ./deploy-web.yaml
                inputs: { version: "${inputs.version}", service: "web" }
          - name: batch-b
            target: web-b
            steps: [...]
```

**语义**：

- `step.workflow` 与 `step.command / step.recipe` **三选一**（Validate 强制）
- 值支持三种形态：
    1. **相对路径**：`./deploy-web.yaml` / `subdir/foo.yaml` —— 相对当前 workflow 文件所在目录解析
    2. **注册引用**：`ref://<id>` —— 走 v0.6.1 Draft/Published 后的 Registry；v0.6.0 先只支持相对路径，`ref://` 保留为**Validate 期错误 `WORKFLOW_REF_UNSUPPORTED`**
    3. **绝对路径**：显式拒绝（防止 `/etc/passwd` 类误用），报 `WORKFLOW_ABS_PATH_FORBIDDEN`
- **递归深度限制**：`MaxSubWorkflowDepth = 5`（硬编码，防栈溢出与循环引用）；超限 → `SUBWORKFLOW_DEPTH_EXCEEDED`
- **循环检测**：Loader 阶段构建"路径栈"，同一路径二次出现 → `SUBWORKFLOW_CYCLE`
- **输入映射**：`step.inputs` 传递给子 workflow 的 `inputs`；子 workflow 校验缺失字段照旧
- **输出映射**：子 workflow 结束时，把最后一步的 `out` 挂到父 `steps.<id>.out.*`（保持与 command / recipe 同构）
- **失败传播**：子 workflow 失败 → 父 step 也失败（`ErrorCode = "SUBWORKFLOW_FAILED"`，`Details.sub_run_id` 指向子 run 的 History Record）；父 workflow 的 rollback 语义不受影响
- **审计与历史**：子 workflow 单独一份 History Record，`Record.ParentRunID` 指向父 run；父 History `Steps[i].SubRunID` 反向索引
- **变量作用域**：子 workflow 的 `${inputs.*}` 只能看到父 step 传入的 `step.inputs`；**不能**看父的 `${steps.*.out.*}`（避免隐式耦合）
- **Timeout**：`step.timeout` 是**整个子 workflow** 的总时限（不是每个子 step 的）

**为什么不做 `include:` 展开而做真正的子 workflow 调用**：

- `include:` 在语法上简单但会引发变量作用域混乱（子 include 能看父的所有 out 吗？）
- 真正的子调用有清晰的 inputs/outputs 边界，与函数调用同构
- History 里能形成父子链路，便于审计追溯

### 3.5 `inputs[].schema:` + 弃用 `inputs[].type`

**动机**：v0.2 的 `inputs[].type ∈ {string,int,bool,file}` 过窄，无法表达 `port ∈ [1,65535]` / `env ∈ enum` / `hosts: string[]` 等常见需求。v0.5.2 已经有成熟的 [core/plugin/settings/schema.go](../core/plugin/settings/schema.go) Schema 编译器，直接复用。

**YAML**：

```yaml
workflow:
  id: deploy.app
  inputs:
    - name: version
      required: true
      schema:
        type: string
        pattern: "^v[0-9]+\\.[0-9]+\\.[0-9]+$"
        description: "Release tag, e.g. v1.2.3"

    - name: port
      required: true
      schema:
        type: integer
        minimum: 1
        maximum: 65535
        default: 8080

    - name: env
      schema: { type: string, enum: [dev, staging, prod], default: dev }

    - name: hosts
      schema:
        type: array
        items: { type: string }
        minItems: 1

    - name: db_password
      schema: { type: string, secret: true }   # 复用 v0.5.2 secret 关键字
```

**兼容性**：

- 保留 `inputs[].type` **作为 shorthand**（v0.6.0 起标 deprecated，v0.7 移除）：

    | shorthand | 等价 schema |
    |---|---|
    | `type: string` | `{ type: string }` |
    | `type: int` | `{ type: integer }` |
    | `type: bool` | `{ type: boolean }` |
    | `type: file` | `{ type: string, format: "path" }` |

- Loader 遇到 `type` 与 `schema` 同时出现 → `INPUT_TYPE_SCHEMA_CONFLICT`
- `default` 顶层字段与 `schema.default` 同时出现 → 取顶层（保持 v0.3 语义），发一个警告

**默认值与校验时机**：

- Loader 阶段：编译 `schema` → `settings.CompiledSchema`（复用现有实现）
- Runner 启动前：合并 CLI/UI 传入的 inputs → `ApplyDefaults` → `Validate` → 全部通过才进入主循环
- 校验失败 → 错误码 `INPUTS_INVALID`，`Details.errors[]` 列出全部违反字段（与 v0.5.2 保持一致，便于 Desktop 表单红标）

**Secret inputs**：

- `schema.secret: true` 的 input 会：
    - **不进入** History Record 的 `Inputs` 字段（复用 v0.4 递归脱敏 [core/command/redact.go](../core/command/redact.go)）
    - **不进入** 结构化日志
    - 传给 Command 时正常明文（Command Engine 有自己的 sensitive 参数机制）

### 3.6 `workflow.idempotency_key:` 幂等键

**动机**：定时触发（v0.6.3）与 Webhook 触发（v0.6.3）会遇到"同一事件重复触发"，需要 Runner 层拒绝重复执行。

**YAML**：

```yaml
workflow:
  id: nightly.backup
  idempotency_key: "${inputs.date}-${inputs.host}"
  inputs:
    - name: date
    - name: host
```

**语义**：

- `idempotency_key` 是一个**表达式模板**（与 `params` 内 `${}` 同款），Runner 启动前求值一次
- 求值后的字符串 + `workflow.id` 组成唯一键
- **默认策略**：`IdempotencyMode = "off"`，即使声明了 key 也不生效（v0.6.0 只做"框架"）
- 未来 v0.6.3 触发器会传 `IdempotencyMode = "reject" | "return-cached"`：
    - `reject`：History 里已有相同 key + 24h 内 → 直接返回错误 `WORKFLOW_DUPLICATE`
    - `return-cached`：直接返回上次 Record，不启动 Runner
- v0.6.0 只做：Loader 校验语法 + Runner 求值并写入 `Record.IdempotencyKey`（供 v0.6.3 使用）
- 空表达式 / 求值为空字符串 → `IDEMPOTENCY_KEY_EMPTY`
- 求值失败 → `IDEMPOTENCY_KEY_INTERPOLATE`

**为什么现在就落地 key，而不是等 v0.6.3**：

- 语法先冻结，避免未来 v0.6.3 tag 一起改 DSL；DSL 变更代价高（需要 workflow 迁移）
- History Record 从 v0.6.2 开始就要新增 `IdempotencyKey` 字段，先在 v0.6.0 引入避免 SQLite schema 再变

## 4. 兼容性影响分析

### 4.1 兼容性矩阵

### 4.3 破坏性变更清单（v0.6.0 的**唯一**破坏点）

**无**。所有变化都是"新字段可选 + 老字段兼容"。

**Deprecation（不阻断）**：

- `inputs[].type` shorthand 会输出 `WARN` 日志：`inputs[].type is deprecated; migrate to schema:{...}`
- v0.7 计划移除；v0.6.x 全系列保留兼容

### 4.4 迁移策略

- **不主动迁移**：官方示例 `examples/workflows/deploy-static-site.yaml` 保留 shorthand，作为兼容性证明
- **提供工具**：新增 `mow workflow migrate <file>` 命令，把 `type:` 展开为 `schema:{...}`；带 `--dry-run` 打印 diff（可选，v0.6.0 P4 交付）
- **文档**：`docs/workflow.md` 新增 §9「v0.6.0 迁移指南」

### 4.5 v0.5.x 承诺的兼容窗口

按 [development-plan §10.2](./development-plan-v0.5-v1.0.md#102-兼容性) 的兼容矩阵：

- Core × Plugin Protocol：v0.6.0 保持不变
- Config schema：v0.6.0 引入 `Config.Workflow.SubWorkflowMaxDepth` 与 `Config.Workflow.IdempotencyWindow`；缺失走安全默认值（5 / 24h）
- Workflow DSL 版本：v0.6.0 引入 `manifestVersion: 2`（可选，缺省视为 1）；`Loader` 明确报告用的哪一版

## 5. Loader / Runner 改造点（实现视角）

以下仅列出**接口层面**变化，具体实现留给对应 PR。

### 5.1 [core/workflow/loader.go](../core/workflow/loader.go)

- `yamlStep` 追加 `ParallelLimit int` / `Target string` / `Workflow string` / `Inputs map[string]any` / `ParallelGroup *yamlParallelGroup`
- `yamlWorkflow` 追加 `IdempotencyKey string` / `ManifestVersion int`
- `yamlInput` 追加 `Schema *yamlSchemaRaw`；`Type` 保留但标 deprecated 处理
- 新增 `LoadWithBase(path, baseDir)` —— 子 workflow 解析用；旧 `LoadFile / LoadBytes / LoadReader` 保持

### 5.2 [core/workflow/workflow.go](../core/workflow/workflow.go)

- `Step` 追加 `ParallelLimit int` / `Target string` / `Workflow *SubWorkflow` / `SubWorkflowInputs map[string]any`
- 新增 `SubWorkflow struct { Path string; Ref string; Loaded *Workflow }` —— Loader 阶段就要 resolve
- `Input` 追加 `Schema *settings.CompiledSchema`
- `Workflow` 追加 `IdempotencyKey string` / `ManifestVersion int`
- `Validate()` 补：
    - `PARALLEL_LIMIT_WITHOUT_PARALLEL`
    - `SUBWORKFLOW_CYCLE` / `SUBWORKFLOW_DEPTH_EXCEEDED`
    - `INPUT_TYPE_SCHEMA_CONFLICT`
    - branch 之间 out 互引拒绝（扩展 [findSiblingRef](../core/workflow/workflow.go#L442) 到 branch 边界）
    - `workflow.command / recipe / workflow` 三选一

### 5.3 [core/workflow/runner.go](../core/workflow/runner.go)

- `RunnerOptions` 追加 `SubWorkflowMaxDepth int`（默 5）与 `IdempotencyStore IdempotencyStore`（默 nil）
- 新增 `interface IdempotencyStore { CheckAndReserve(key) (locked bool, err error); Release(key) }` —— v0.6.0 提供 in-memory 实现，SQLite 版走 v0.6.2
- `runStep` 新增分支：`step.Workflow != nil` → `runSubWorkflow(ctx, step, scope)`
    - 派生独立 `RunID`，`ParentRunID = 外层 RunID`
    - History Record 独立写盘；父 step 的 `Result.SubRunID` 保留反向索引
- 并行组：`groupSemaphore` 走 `parallel_limit`；`parallel_group.branches[]` 走单独 goroutine per branch
- `runGroup` 递归深度 = 1（v0.6.0 限制）

### 5.4 [core/workflow/vars.go](../core/workflow/vars.go)

- Interpolate 新增函数入口 `InterpolateIdempotencyKey(scope, key) (string, error)`
- 子 workflow 的 `scope` 是**新的**（`inputs = 父传入的 step.inputs`，`steps = {}`），保证隔离

### 5.5 History Record 演进

```go
type Record struct {
    // 现有字段...
    ParentRunID     string                  `json:"parent_run_id,omitempty"`
    SubRunID        string                  `json:"sub_run_id,omitempty"`     // 父 step 视角
    IdempotencyKey  string                  `json:"idempotency_key,omitempty"`
    ManifestVersion int                     `json:"manifest_version,omitempty"`
}
```

**JSONL 向后兼容**：旧行缺这几个字段 → JSON decode 走零值；`List / Get` 输出正常。

## 6. 错误码新增清单

按 [development-plan §10.4](./development-plan-v0.5-v1.0.md#104-可观测性) 的稳定错误码约束，v0.6.0 引入以下**新**错误码（前缀均为 `WORKFLOW_` 或字段名）：

| 错误码 | 触发点 | 重试性 |
|---|---|---|
| `PARALLEL_LIMIT_WITHOUT_PARALLEL` | Validate | 不重试（用户错误） |
| `PARALLEL_LIMIT_EXCEEDS_CAP` | Validate（> 512） | 不重试 |
| `SUBWORKFLOW_CYCLE` | Loader | 不重试 |
| `SUBWORKFLOW_DEPTH_EXCEEDED` | Runner | 不重试 |
| `SUBWORKFLOW_FAILED` | Runner | 由子 workflow 错误决定 |
| `WORKFLOW_REF_UNSUPPORTED` | Loader | 不重试（v0.6.0 保留位） |
| `WORKFLOW_ABS_PATH_FORBIDDEN` | Loader | 不重试 |
| `INPUT_TYPE_SCHEMA_CONFLICT` | Loader | 不重试 |
| `INPUTS_INVALID` | Runner 启动前 | 不重试（`Details.errors[]`） |
| `IDEMPOTENCY_KEY_EMPTY` | Runner | 不重试 |
| `IDEMPOTENCY_KEY_INTERPOLATE` | Runner | 不重试 |
| `WORKFLOW_DUPLICATE` | v0.6.3 触发器；v0.6.0 保留位 | 不重试 |
| `BRANCH_OUT_REF` | Validate | 不重试 |

**保持不变**：`WHEN_EVAL` / `INTERPOLATE` / `NO_EXECUTOR` / `INVALID_STEP` / `STEP_FAILED`。

## 9. 未决问题（Open Questions）

以下问题不阻塞 v0.6.0 起草落地，但要在实现前敲定：

1. **子 workflow 的 `on_failure.rollback`**：子失败时，子的 rollback 先跑，还是直接跳过？
    - 倾向：**子自己先 rollback**，然后把失败传给父；父的 rollback 只对父级 step 生效
    - 待与 v0.9 AI Plan 场景交叉验证

2. **子 workflow 的 `parallel_limit` 是否与父共享？**
    - 倾向：**独立**。父跑 3 个 sub，每个 sub 自己有 `parallel_limit=8`，总并发 24
    - 如果要共享 → 需要引入 "Runner-wide concurrency budget"，太复杂，v0.7 再谈

3. **`step.workflow` 传参用 `inputs` 还是 `params`？**
    - 倾向：`inputs`（与 Workflow 顶层字段同名，语义一致）
    - `params` 保留给 command/recipe，不混用

4. **Idempotency 窗口默认值**
    - 倾向：24h。触发器场景足够，Runbook 手动重放通常不会超过一天
    - 可通过 `Config.Workflow.IdempotencyWindow` 覆盖

5. **`branches[].name` 是否强制唯一？**
    - 倾向：**强制唯一**。用于 CLI / Desktop 日志区分
    - 冲突 → `BRANCH_NAME_DUPLICATE`

6. **`inputs[].schema.default` 与 `inputs[].default` 冲突时的日志级别**
    - 倾向：`WARN`（不阻断）；v0.7 提升到 `ERROR`
