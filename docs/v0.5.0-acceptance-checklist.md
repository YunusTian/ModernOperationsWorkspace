# v0.5.0 验收清单 — Plugin Manifest 与包格式

- 状态：Implemented（P1–P5 已交付；§7 发布门槛待 Release CI）
- 目标版本：v0.5.0
- 更新日期：2026-07-11
- 范围：只做 Manifest schema、`mow plugin validate` 命令、运行时对 Manifest 的强校验。**不做**下载 / 安装 / Catalog / 配置 UI（这些属于 v0.5.1 / v0.5.2）。
- 相关文档：[开发计划 §4.2](./development-plan-v0.5-v1.0.md#42-v050plugin-manifest-与包格式) · [Plugin System RFC](./plugin-system.md#6-plugin-manifestv050)

## 1. Manifest 规范与 Schema

- [x] `sdk/manifest/plugin.schema.json` 作为 Manifest 的权威 JSON Schema（draft 2020-12）
- [x] Schema 覆盖字段：`id` / `name` / `version` / `author` / `license` / `homepage` / `description`
- [x] Schema 覆盖兼容范围：`compatibility.core` / `compatibility.sdk` / `compatibility.protocol`（语义化 range 字符串）
- [x] Schema 覆盖平台矩阵：`platforms[]`，每项含 `os` / `arch` / `entrypoint` / `checksum`
- [x] Schema 覆盖能力摘要：`commands[]`（id / permission）、`connectionTypes[]`、`permissions[]`
- [x] Schema 覆盖 `settingsSchema`（内嵌 JSON Schema，v0.5.2 才驱动 UI，v0.5.0 只校验其合法性）
- [x] Schema 覆盖资源清单：`recipes[]` / `workflows[]`（相对包内路径）
- [x] Schema 覆盖数据版本与迁移入口：`dataVersion` / `migrations[]`
- [x] Schema 覆盖发布来源与签名：`source.url` / `source.tag` / `signature.algorithm` / `signature.value`（v0.5.0 允许缺省）
- [x] Schema 有单元测试，覆盖合法 / 非法样本，错误码稳定（19 个失败分支 + happy path）

## 2. Manifest 加载与解析

- [x] `sdk/manifest` 提供 `Load(path) (Manifest, error)` / `Parse(data) (Manifest, error)`
- [x] 解析失败返回 `PLUGIN_MANIFEST_INVALID` + 定位（`Details.field` 精确到 `platforms[2].checksum` 粒度）
- [x] 兼容范围解析基于 semver constraint（如 `>=0.5.0,<0.6.0`）；零依赖手写求解器，pre-release 顺序遵循 SemVer 2.0.0
- [x] Manifest 与 gRPC Metadata 的字段映射有单向 diff 工具（`Manifest.MatchMetadata(sdk.Metadata)`）
- [x] SDK 契约测试覆盖 Manifest 反序列化、兼容范围解析、错误码稳定性（`sdk/manifest` 覆盖率 81.4%）
- [x] 未新增破坏性 SDK 或 Plugin Protocol 变更（Manifest 仅补充，运行时协议不变）

## 3. `mow plugin validate` 命令

- [x] `mow plugin validate <path>` 支持传入 `plugin.json` 或包目录
- [x] 输出人类可读报告 + `--json` 结构化输出
- [x] 稳定错误码：`PLUGIN_MANIFEST_INVALID` / `PLUGIN_CHECKSUM_MISMATCH` / `PLUGIN_ENTRYPOINT_MISSING`
- [x] 校验 checksum：对每个 `platforms[].entrypoint` 计算 SHA-256 并与 Manifest 声明比对
- [x] 校验 `recipes[]` / `workflows[]` 引用的相对路径实际存在
- [x] `-v` verbose 模式打印所有通过项，便于插件开发者调试
- [x] 单元测试覆盖 happy path + 五类失败分支（7 个 CLI 用例 + 6 个包校验用例）

## 4. 运行时强校验

- [x] Plugin Manager 在 Load 阶段读取包目录的 `plugin.json`（`LoadFromPackage` / `Manager.RegisterFromPackage`）
- [x] Manifest.ID / Version 与 gRPC `sdk.Metadata` 返回值不一致 → 拒绝启用，错误码 `PLUGIN_MANIFEST_MISMATCH`
- [x] `compatibility.core` 不满足当前 Core 版本 → 拒绝加载，错误码 `PLUGIN_INCOMPATIBLE`
- [x] `compatibility.sdk` / `compatibility.protocol` 不满足 → 同错误码
- [x] 拒绝发生在启动子进程**之前**（`CheckCompatibility` 关卡不调用 `loadBinary`，测试用例专项覆盖）
- [x] 拒绝原因附带具体字段与期望值（`Details.layer` / `actual` / `constraint` / `field`），便于诊断
- [x] 单元测试覆盖 Mismatch / Incompatible / Happy 三条主路径（8 个 gate 用例，`core/plugin` 覆盖率 81.8%）

## 5. 官方插件补齐 plugin.json

- [x] `plugins/ssh/plugin.json` 通过静态 validate（本地 `bin/` 未构建时 checksum 校验期望失败）
- [x] `plugins/docker/plugin.json` 通过静态 validate
- [x] `plugins/ai/plugin.json` 通过静态 validate
- [x] 三份 Manifest 中的 commands / permissions 与运行时 Metadata 一一对齐（`manifest_test.go` 强校验）
- [ ] Release Workflow 打包时把 `plugin.json` 一同放入 tar.gz（v0.5.0 发布阶段完成）
